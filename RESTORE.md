# Restoring a Ledger DR Backup to a Local Postgres + PostGIS

This document explains how to take the ZIP produced by **Settings → Backup**
inside the live Ledger DR app and restore it to a plain **PostgreSQL + PostGIS**
database on your own machine.

The result is a **cold, read-only mirror** of the production data — useful as
an offline backup, for ad-hoc queries, or for auditing — without any
dependency on Lovable Cloud.

---

## 1. What this is (and isn't)

**Is**

- A queryable copy of every business table (transactions, journals, employees,
  inventory, fuel, operations, etc.).
- All NCF and HR attachments on disk.
- Idempotent: re-running newer ZIPs tops up the data without breaking what's
  already there.

**Isn't**

- A live, writable mirror. The Ledger DR React app cannot be pointed at this
  database — it expects Supabase Auth, RLS, edge functions, storage and
  realtime, none of which exist on plain Postgres.
- A replacement for the production system. Treat it as a vault.

To browse it you use **`psql`** or any Postgres GUI
(**DBeaver** is recommended — free, cross-platform, handles PostGIS geometry).

---

## 2. What's inside the backup ZIP

The ZIP is generated by `src/components/settings/DatabaseBackup.tsx`. Layout:

```text
ledgerdr-backup-YYYY-MM-DD.zip
├── 00_schema.sql        # tables, enums, helper functions
├── 01_data.sql          # INSERT … ON CONFLICT DO NOTHING (idempotent)
├── 02_rls_policies.sql  # SKIP on plain Postgres (uses auth.uid / has_role)
├── 03_triggers.sql      # load it; ignore errors on auth-dependent triggers
├── 04_storage.sql       # SKIP (Supabase storage schema only)
├── backup.json          # full data dump in JSON form
├── tables/              # one JSON file per table
│   └── *.json
├── attachments/
│   ├── transactions/    # NCF files (PDF / JPG)
│   └── employees/       # HR documents
├── metadata.json        # export summary and statistics
└── README.md            # auto-generated summary of this export
```

## 3. Prerequisites

On the laptop / server that will hold the backup:

- **PostgreSQL 15 or newer** with the **PostGIS** extension package installed.
- **`psql`** on your `PATH`.
- **`unzip`** (or any ZIP utility).
- A GUI client — recommended: **[DBeaver](https://dbeaver.io/)** (free).
  Alternatives: pgAdmin, TablePlus, Postico.

Quick install hints:

| OS              | Command                                                                 |
|-----------------|-------------------------------------------------------------------------|
| macOS (Homebrew)| `brew install postgresql@16 postgis`                                    |
| Ubuntu / Debian | `sudo apt install postgresql-16 postgresql-16-postgis-3`                |
| Windows         | Use the EnterpriseDB installer + the StackBuilder PostGIS bundle        |

## 4. One-time setup

Create the database, install the extensions, and stub out the Supabase-only
`auth.uid()` / `auth.role()` functions so any function or trigger that
references them still loads cleanly.

```sh
createdb ledgerdr_backup

psql ledgerdr_backup -c "CREATE EXTENSION IF NOT EXISTS postgis;"
psql ledgerdr_backup -c "CREATE EXTENSION IF NOT EXISTS \"uuid-ossp\";"
psql ledgerdr_backup -c "CREATE EXTENSION IF NOT EXISTS pgcrypto;"

psql ledgerdr_backup <<'SQL'
CREATE SCHEMA IF NOT EXISTS auth;

CREATE OR REPLACE FUNCTION auth.uid() RETURNS uuid
  LANGUAGE sql STABLE AS $$ SELECT NULL::uuid $$;

CREATE OR REPLACE FUNCTION auth.role() RETURNS text
  LANGUAGE sql STABLE AS $$ SELECT 'postgres'::text $$;

CREATE OR REPLACE FUNCTION auth.jwt() RETURNS jsonb
  LANGUAGE sql STABLE AS $$ SELECT '{}'::jsonb $$;
SQL
```

You only do this once per database.

## 5. Loading a backup ZIP

```sh
# 1. Unzip
unzip ledgerdr-backup-YYYY-MM-DD.zip -d ./backup
cd backup

# 2. Schema first
psql ledgerdr_backup -v ON_ERROR_STOP=0 -f 00_schema.sql

# 3. Data
psql ledgerdr_backup -v ON_ERROR_STOP=0 -f 01_data.sql

# 4. Triggers (some reference auth — errors there are expected and harmless)
psql ledgerdr_backup -v ON_ERROR_STOP=0 -f 03_triggers.sql

# DO NOT run these on plain Postgres — they target Supabase-only schemas:
#   02_rls_policies.sql   (auth.uid / has_role)
#   04_storage.sql        (storage.objects, storage.buckets)
```

Notes:

- **`ON_ERROR_STOP=0`** lets the load continue past the few statements that
  reference Supabase-only objects. Skim the output for unexpected errors —
  per-statement failures on `auth.users`, `storage.*`, `realtime.*` are
  expected and can be ignored.
- **`01_data.sql` is idempotent.** Re-running it after a fresher export will
  add new rows and leave existing rows untouched (`ON CONFLICT DO NOTHING`).
  It will **not** update changed rows — for that you'd need a fresh database
  or a custom merge.

## 6. Restoring attachments

Move the `attachments/` folder somewhere stable, e.g.:

```sh
mkdir -p ~/ledgerdr-backup
mv attachments ~/ledgerdr-backup/attachments
```

Resulting layout:

```text
~/ledgerdr-backup/attachments/
├── transactions/   # NCF files referenced by transaction_attachments
└── employees/      # documents referenced by employee_documents
```

The `attachment_url` columns in the database still point at the **cloud**
URL. To find the local file, strip the cloud prefix and prepend your local
folder. Example URL stored in the DB:

```text
https://hirrjrgakqlkmrrxabnv.supabase.co/storage/v1/object/public/transaction-attachments/<path>
```

corresponds to:

```text
~/ledgerdr-backup/attachments/transactions/<path>
```

Optional helper view that exposes a `local_path` column for convenience:

```sql
CREATE OR REPLACE VIEW transaction_attachments_local AS
SELECT
  ta.*,
  '/Users/you/ledgerdr-backup/attachments/transactions/' ||
    regexp_replace(
      ta.attachment_url,
      '^.*/transaction-attachments/',
      ''
    ) AS local_path
FROM transaction_attachments ta;
```

(Adjust the path prefix to match your machine.)

## 7. Browsing the data

### Command line

```sh
psql ledgerdr_backup
\dt                                       -- list tables
\d transactions                           -- describe one
SELECT count(*) FROM transactions;
\copy (SELECT * FROM journal_entries
       WHERE entry_date >= '2026-01-01')
  TO 'jan-journals.csv' CSV HEADER
```

### GUI (DBeaver, pgAdmin, …)

Connect with:

| Field    | Value                |
|----------|----------------------|
| Host     | `localhost`          |
| Port     | `5432`               |
| Database | `ledgerdr_backup`    |
| User     | your local Postgres user |

You then get spreadsheet-style table browsing, query editor with autocomplete,
CSV/Excel export, and ER diagrams — no custom code needed.

### Cheat sheet — common queries

```sql
-- Latest 50 transactions
SELECT transaction_date, transaction_type, name, amount, currency
FROM transactions
ORDER BY transaction_date DESC
LIMIT 50;

-- Trial balance from posted journal lines
SELECT a.code, a.name,
       SUM(jl.debit)  AS debit,
       SUM(jl.credit) AS credit,
       SUM(jl.debit - jl.credit) AS balance
FROM journal_lines jl
JOIN journal_entries je ON je.id = jl.journal_entry_id
JOIN accounts a         ON a.id  = jl.account_id
WHERE je.is_posted = true
GROUP BY a.code, a.name
ORDER BY a.code;

-- Payroll summary for a period
SELECT period_start, period_end,
       SUM(net_pay) AS net_total
FROM payroll_runs
GROUP BY 1,2
ORDER BY 1 DESC;

-- Transactions missing an NCF attachment
SELECT t.id, t.transaction_date, t.name, t.amount
FROM transactions t
LEFT JOIN transaction_attachments ta
  ON ta.transaction_id = t.id AND ta.attachment_category = 'ncf'
WHERE t.transaction_type = 'purchase'
  AND ta.id IS NULL
ORDER BY t.transaction_date DESC;
```

## 8. Refreshing the backup

Suggested cadence: **weekly**.

```sh
# 1. In the live app: Settings → Backup → download ZIP
# 2. On the laptop:
unzip -o ledgerdr-backup-YYYY-MM-DD.zip -d ./backup
cd backup

# Re-run loads (safe to repeat)
psql ledgerdr_backup -v ON_ERROR_STOP=0 -f 00_schema.sql
psql ledgerdr_backup -v ON_ERROR_STOP=0 -f 01_data.sql

# Sync attachments (only adds new files; existing files preserved)
rsync -av attachments/ ~/ledgerdr-backup/attachments/
```

- `00_schema.sql` uses `CREATE TABLE IF NOT EXISTS`, so new tables / columns
  added in production over time will appear in your local copy on the next
  refresh; existing tables are left as-is.
- `01_data.sql` only inserts new rows. **Updates and deletes in production are
  not reflected** in your local copy — it's an append-only mirror. For a
  fully-current snapshot, drop the database and reload from scratch:
  ```sh
  dropdb ledgerdr_backup && createdb ledgerdr_backup
  # then repeat sections 4 and 5
  ```

## 9. What is *not* restored, and why

| Not restored        | Reason                                                   |
|---------------------|----------------------------------------------------------|
| `auth.users`, sessions, password hashes | Managed by Supabase Auth — not in the export. |
| RLS policies        | Reference `auth.uid()` and `has_role()`; meaningless on plain Postgres. |
| Storage bucket policies | Reference `storage.objects` schema (Supabase-only). |
| Edge function secrets | Stored in Lovable Cloud secrets — keep these in a password manager. |
| Realtime publications | Require the `supabase_realtime` extension and replication slots. |
| Live signed URLs    | The signed-URL machinery is Supabase-side; use the local files instead. |

## 10. If you ever want a *live* local copy

This document covers cold backups only. Running the full Ledger DR app
against a local stack requires the Supabase services (GoTrue, PostgREST,
Storage API, Realtime). Two starting points:

- **Self-hosted Supabase**: <https://supabase.com/docs/guides/self-hosting>
- **DigitalOcean migration blueprint**: see project memory
  `mem://technical/migration/digitalocean-blueprint` for the planned move
  from Supabase to a Ubuntu Droplet.

Both are larger projects — out of scope for a backup workflow.
